# 开发工作流(全局约定)

以下规则适用于所有项目,严格遵守。

## 总原则

- 未经我批准,不写实现代码。先理解、先分析、先给方案。
- `main` / `master` 是受保护分支:不直接提交、不合并、不推送到主分支。合并到主分支由我本人在 GitHub 上执行。
- 功能分支(`feat/*`、`fix/*`)可以提交、推送、开 PR——这是交付的标准方式。
- 每个任务用独立的功能分支。
- 不确定就问,不要假设着往下做。

## 标准流程

1. **理解需求**:接到需求先用一段话复述你的理解,列出边界与假设;有不明确的点先向我提问。需求不清时不要猜着写代码。

2. **分析 / 定位**(此阶段不写实现代码):
   - 新功能:给出实现方案——涉及哪些文件/模块、关键数据结构或接口、边界情况、潜在风险、测试要点。不要贴大段实现代码。
   - 修 bug:先找**根因**,沿调用链/数据流定位到根本原因,说明为什么会发生、影响范围,再提修复方案。区分"根因"与"表象",不要只治标。

3. **等我批准**:把"需求理解 + 方案"发我审核。我明确说"可以 / 开始"之后,才进入开发。

4. **拉分支开发**:从最新的 `main` 拉功能分支(`feat/xxx`、`fix/xxx`),在分支上小步提交。

5. **自检**:开发完运行测试 / lint / 构建,自己先过一遍,修掉问题。

6. **交付待审(PR 流程)**:在功能分支上 `git commit` → `git push` 到 origin → 用 `gh pr create` 开 PR(base=main)。PR 正文写清:变更摘要、自检结果、部署前置(如 DB/表结构变更)。然后**停下**。不要合并、不要推送到 main。我在 GitHub 上审核并合并。后续若按我意见改动,提交并推同一分支,PR 自动更新。

## 硬性禁止(会被 hook 拦截)

- 推送涉及 `main` / `master` 的操作(含 `git push origin main`、`... HEAD:main`)
- `git merge`、`git reset --hard`、`git push --force` / `-f`
- 在 `main` / `master` 上 `git commit`
- 删除 `main` / `master` 分支

**允许**:在功能分支(非 main/master)上 `git commit`、`git push`、`gh pr create`。

被拦截时:不要重试、不要绕过、不要改用别的命令达到同样目的。停下,告诉我,由我处理。

## 网络访问(白名单机制)

- 本环境的外网出口是**白名单机制**,默认很多域名不可达(表现为 `no route to host` / 连接超时)。
- 需要访问某个特定域名时(拉依赖、调外部 API、构建等),**直接把域名告诉我,由我加入白名单**;不要绕路或放弃。此约定适用于所有项目。

## 沟通

- 用中文。
- 每一步简洁说清:你做了什么、下一步需要我做什么。
